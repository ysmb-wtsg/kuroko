# ADR 0008: 外部エディタダイアログの追加（プレビューと併存）

## ステータス

採用（2026-06-15）

## コンテキスト

ファイルツリーの `p` キーは、syntectでシンタックスハイライトした読み取り専用プレビューを
フローティング表示する。確認には十分だが編集はできず、開いたファイルを直すには結局
ターミナルペインで別途エディタを起動する必要があり、操作が分断されていた。

「エディタに代わるホームベース」という目標に対し、ファイルツリーから直接編集に入れる
手段が欲しい。一方で、軽量に中身を確認するだけのプレビューにも固有の価値があり、
こちらも残したい。

検討した代替案:

- **プレビューを廃止してエディタに一本化**: 一度は採用しかけたが、編集を伴わない素早い
  閲覧（読み取り専用・スクロールのみ）の体験はエディタ起動より軽く、別物として価値がある。
  ユーザーの意向によりプレビューは維持する
- **自前の簡易テキストエディタを実装**: 入力モデル・アンドゥ・保存などの再実装が必要で、
  vim/nvim/helix等の慣れたエディタを使いたい要求に応えられない。YAGNI違反
- **新規ターミナルペインとして開く**: 基盤流用は容易だが、「開いて閉じる」一時操作に
  ペイン分割は重く、「ダイアログ」という体験から外れる

## 決定

1. **プレビュー（`p` / `Action::ToggleFilePreview`）はそのまま維持する**
2. **ファイルツリーの `e` キーで外部エディタをフローティングダイアログ内のPTYで起動する**
   （`Action::OpenEditor(PathBuf)`）。プレビューと編集を別キーで併存させる
3. **エディタは既存のPTY基盤（`TerminalPane`）を流用する**。エディタ用ペインを `App::panes` に
   登録し、PTY出力の配送（`drain_pty_messages`）をそのまま再利用する。ダイアログ状態
   （`OverlayState::editor: Option<EditorOverlay>`）は対象の `pane_id` と `path` のみを持つ
4. **エディタの解決順は 設定 `krk.opt.editor` → 環境変数 `$EDITOR` → `vim`**。
   値は空白区切りでプログラムと引数に分割する（例: `"nvim -u NONE"`）
5. **ダイアログ表示中は全キーをエディタPTYへ直通させ、アプリはキーを一切奪わない**。
   ダイアログはエディタプロセスの終了（例: vimの `:q`）で自動的に閉じる。
   `PtyMessage::Exited` 受信時に該当ペインを破棄しオーバーレイを閉じる

## 結果

- `p` で軽量プレビュー、`e` で本格編集、と用途に応じて使い分けられる
- ユーザーは慣れたエディタをそのまま使える（設定/環境変数で指定可能）
- ADR 0007（直通＝アプリはキーを奪わない）と整合する: ダイアログ中もエディタが入力を所有する
- トレードオフ:
  - ダイアログを閉じる手段がエディタ側の終了操作に限られる（アプリ側の強制クローズキーは持たない）。
    これは「エディタが入力を所有する」原則を優先した結果で、グローバルな状態を増やさない方針にも沿う
  - 多重起動は防止する（ダイアログ表示中の `OpenEditor` は無視）
  - プレビュー（syntect依存）とエディタが併存するため、両系統のコードを保守する
