# ADR 0010: エージェント入力待ちのデスクトップ通知（krk.opt.notify）

## ステータス

採用（2026-06-16）

## コンテキスト

エージェントは応答に数十秒〜数分かかることがある。ユーザーは待ち時間に別の
ウィンドウやタブへ離れるため、エージェントが処理を終えて「ユーザーの番」に
なったことに気づけず、待ち時間が無駄に伸びる。

要求された通知タイミングは2つ:

- **タスク完了**: ユーザーの入力待ちになったタイミング
- **確認待ち**: 「1,2,3,4 を選択させる」等のインタラクティブな確認を要求しているタイミング

ただし kuroko のエージェント状態推定（ADR外、`status.rs`）は **プロバイダー固有の
出力パースを行わない** 方針で、最後の出力からの経過時間（600ms）だけで
Working / Idle を推定している。この制約下では「確認待ち」と「完了して入力待ち」は
どちらも出力が途絶えた `Idle` に集約され、両者を区別できない。

検討した代替案:

- **エージェント出力をパースして確認待ちを検出する**: Claude Code / Codex 固有の
  確認プロンプト形式に依存する。Non-negotiables「特定ケース向けハックの禁止」と
  `status.rs` の既存方針に反するため不採用
- **ターミナルベル(BEL)を確認待ちの signal として使う**: 汎用ターミナル仕様で
  方針には反しないが、完了時にもベルを鳴らすエージェントがあり「確認待ち」を
  厳密に切り分けられない。区別の確実性が上がらないため今回は見送る
- **macOS `osascript` で通知する**: 確実だが macOS 専用かつ外部プロセス起動が必要

## 決定

1. **通知の発火条件は `Working→Idle` の状態遷移とする**。確認待ちと完了は区別せず、
   「エージェントの処理が止まりユーザーの番になった瞬間」を1種類の通知でカバーする。
   要求の2タイミングは、どちらも出力が途絶えて `Idle` になるため、この単一の遷移で
   実質的に両方を捉えられる
2. **遷移検出は `IdleNotifier`（`status.rs`）で行う**。直前状態を保持し、
   `Working→Idle` の瞬間だけ true を返す。Idle 継続中は再発火せず、再び Working に
   戻って再び Idle になれば再発火する。時間経過で状態が変わるため、メインループから
   毎フレーム `AgentPane::poll_idle_notification()` でポーリングする
3. **通知は OSC 9（`ESC ] 9 ; <body> BEL`）を外側端末へパススルーする**。
   対応端末（iTerm2 / WezTerm / Ghostty / kitty 等）がOSの通知として表示する。
   OS非依存・外部プロセス不要・SSH越しでも機能し、非対応端末は未知のOSCとして
   無視するため画面表示に影響しない
4. **フォーカス中のペインは通知しない**。ユーザーが今見ているペインの完了は冗長なため
   抑制する。グローバルな状態は増やさず、既存の `focused` と比較するだけにとどめる
5. **`krk.opt.notify`（デフォルト true）でオンオフできる**。git_tool / editor /
   file_manager と一貫した `krk.opt.*` 設定体験。無効時もポーリングして遷移状態は
   更新し、有効化直後の取りこぼし発火を防ぐ

## 結果

- ユーザーは別ウィンドウで作業していてもエージェントの入力待ちに気づける
- プロバイダー固有パースを増やさず、既存の時間ベース状態推定にそのまま乗せた
- グローバルな状態を増やさず、各 `AgentPane` 内部の `IdleNotifier` に閉じている
- トレードオフ:
  - 「確認待ち」と「タスク完了」を区別しない。両者は同じ「入力待ち」通知になる。
    厳密な区別はプロバイダー固有パースを要し、方針に反するため将来課題（未実装）
  - OSC 9 非対応の端末では視覚的な通知が出ない。端末側の対応に依存する

## 改訂（2026-06-16）

決定4「フォーカス中のペインは通知しない」を **撤回** する。

実機（WezTerm）での検証で、単一エージェントペインに指示を送って端末ウィンドウごと
別アプリ（ブラウザ等）へ移ったケースで通知が一切出ないことが判明した。原因は、
アプリ内のペインフォーカス（`focused`）と端末ウィンドウのOSフォーカスが別物である点。
ユーザーが離れていてもアプリ内では唯一のペインが `focused` のままのため、
`id != focused` 条件が常に偽になり通知が握り潰されていた。

これは最も一般的なユースケース（1ペインに指示して離席）で通知が出ないことを意味し、
機能の目的に反する。よって**ペインフォーカスによる抑制は行わない**。端末ウィンドウが
フォーカス中かどうかに基づく通知抑制は、端末側（例: WezTerm の `notification_handling`）に
委ねる。アプリは状態遷移を素直にOSCで送出するだけにする。
