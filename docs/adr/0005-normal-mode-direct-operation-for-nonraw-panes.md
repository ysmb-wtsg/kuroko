# ADR 0005: 非rawペインのNormalモード直接操作

## ステータス

採用（2026-06-13）

## コンテキスト

FileTreeペインの操作（カーソル移動、展開、プレビュー等）はInsertモードに入らないと行えなかった。これには以下の問題があった。

1. **モード意味論の歪み**: Insertモードの本来の意味は「キー入力をPTYに生で送る」（ターミナル/エージェント用）。FileTreeはテキスト入力先ではなく閲覧・操作UIであり、Insertモードを経由する意味的な根拠がない
2. **Neovimの感覚との乖離**: Neovimではファイルツリー（nvim-tree等）はただのバッファで、フォーカスすれば j/k が直接効く。ウィンドウ移動側がプレフィックスキー（Ctrl-w）に譲る
3. **原因**: Normalモードの裸の hjkl がペインフォーカス移動を占有していたため、FileTree内のカーソル移動と衝突し、回避策としてペイン操作がInsertモードに押し込まれていた

## 決定

1. **非rawペイン（`wants_raw_input() == false`、現状はFileTree）にフォーカス中は、Normalモードのキーをペインに直接ルーティングする**。Insertモードを経由しない
2. **グローバル予約キーを最小集合に絞る**。非rawペインフォーカス中もAppが先取りするのは以下のみ:
   - `Ctrl+h/j/k/l`: フォーカス移動（全ペイン種別で常に有効）
   - `Tab` / `S-Tab`: フォーカス順送り
   - `t` / `f` / `g`: パネルトグル
   - `:`: コマンドパレット
   - `q`: 終了
3. **rawペイン（ターミナル/エージェント）フォーカス中の挙動は従来通り**。裸の hjkl はフォーカス移動として維持する（ペイン側がキーを消費しないため衝突がない）
4. **Insertモードはrawペイン専用の概念に戻る**。「キーがPTYに送られるモード」という一貫した意味を持つ

## 結果

- FileTreeフォーカス中、`i` はInsertモード切替ではなくペインのファイル情報表示になる（非rawペインにInsertモードは不要のため許容）
- FileTreeフォーカス中、`H`（隠しファイル切替）がペインに渡るため、リサイズの `H/J/K/L` は使えない。リサイズはrawペインフォーカス時のみ
- ペインキーの衝突解決は「予約キーをAppが先取り→残りは全てペインへ」の単純な優先順位で行い、`handle_event` の戻り値による消費判定（trait変更）は導入しない。ペイン側が予約キーを使いたくなった時点で再検討する
